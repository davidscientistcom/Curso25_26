Configuración Avanzada de Linters

Definiendo el "Contrato de Estilo": Dónde, Cómo y Qué se puede configurar.

El "Dónde": Estructura del Proyecto

Los ficheros de configuración son "artefactos de gobernanza" y viven en la **raíz (root)** del repositorio.

¿Por qué en la raíz?

  • Es el primer lugar donde un desarrollador (o un sistema de CI/CD) los busca.
  • Aplican de forma global a todo el proyecto (ej. `src/` y `tests/`).

El estándar moderno de Python (PEP 518) consolida toda la configuración en un solo fichero: pyproject.toml.

mi_proyecto/
├── .git/
├── .venv/
├── src/
│   └── mi_app/
│       └── __init__.py
├── tests/
│   └── test_mi_app.py
│
# --- Lugar de Configuración ---
├── pyproject.toml  # (El Estándar Moderno)
│
# --- Opciones Legado (A evitar si es posible) ---
├── .flake8
├── .pylintrc
├── mypy.ini
└── README.md

El "Cómo": Unificando en pyproject.toml

En lugar de tener 5 ficheros de configuración (`.flake8`, `.pylintrc`, `mypy.ini`, `isort.cfg`...), el ecosistema de Python ha convergido en `pyproject.toml`.

Cada herramienta lee su propia sección, identificada con [tool.NOMBRE_HERRAMIENTA].

Ventajas

  • **Un solo fichero:** Mantiene la raíz del proyecto limpia.
  • **Estándar:** Soportado por todas las herramientas modernas.
  • **Formato TOML:** Más legible que `setup.cfg` o `.ini`.
# Fichero: pyproject.toml

# Black (Formateador)
[tool.black]
line-length = 88
target-version = ['py310']

# isort (Imports)
[tool.isort]
profile = "black"
line_length = 88

# flake8 (Linter)
[tool.flake8]
max-line-length = 88
ignore = "E203, W503"

# mypy (Tipos)
[tool.mypy]
python_version = "3.10"
ignore_missing_imports = true

A Fondo (flake8): Plugins y Exclusiones

flake8 es un "meta-linter". Su poder real viene de sus plugins (ej. `flake8-bugbear`, `flake8-docstrings`).

La configuración nos permite ser granulares, incluso ignorando reglas para ficheros específicos (como los `__init__.py`).

# Fichero: pyproject.toml
[tool.flake8]
max-line-length = 88
max-complexity = 12
ignore = "E203, W503"

# Configurar plugins: (ej. flake8-docstrings)
# Forzar a que todas las funciones públicas tengan docstring
docstring-convention = "google"

# Configurar plugins: (ej. flake8-bugbear)
# B008: No permitir llamadas a funciones como valores por defecto
extend-ignore = "B008" 

# --- Configuración por Fichero ---
# Ignorar el error F401 (import no usado) en los ficheros __init__
# (útil para exponer imports del paquete)
[tool.flake8."per-file-ignores"]
"src/mi_app/__init__.py" = "F401"
"tests/*" = "D100, D103" # Ignorar "missing docstring" en tests

A Fondo (pylint): Definiendo Estilo (Nivel StyleCop)

pylint es el más configurable. Responde directamente a su pregunta sobre "definir un estilo" mediante expresiones regulares (Regex) para el nombrado.

Esto es un contrato de gobernanza explícito.

# Fichero: .pylintrc (o pyproject.toml)

[tool.pylint.main]
disable = """
    missing-docstring,
    invalid-name, # Desactivamos 'invalid-name' genérico...
    """

# ...para definir el nuestro!
[tool.pylint.basic]
# Forzar "snake_case" para variables (ej. mi_variable)
variable-rgx = "[a-z_][a-z0-9_]{2,30}$"

# Forzar "PascalCase" para clases (ej. MiClase)
class-rgx = "[A-Z_][a-zA-Z0-9]+$"

# Forzar "snake_case" para funciones (ej. mi_funcion)
function-rgx = "[a-z_][a-z0-9_]{2,30}$"

# Permitir nombres cortos comunes en bucles o excepciones
good-names = "i, j, k, f, e, db, id"

# Forzar un máximo de argumentos en una función
max-args = 5

A Fondo (mypy): Forzando el Tipado Estricto

mypy puede configurarse para ser extremadamente estricto, similar a un lenguaje compilado. Esto es crucial para proyectos grandes.

El modo strict = true activa todas las comprobaciones de "buenas prácticas".

# Fichero: pyproject.toml

[tool.mypy]
python_version = "3.10"
strict = true # ¡Activa todas las comprobaciones estrictas!

# --- O de forma granular ---

# Falla si una función no tiene type hints
disallow_untyped_defs = true

# Falla si una función es llamada con tipos 'Any'
disallow_any_generics = true

# Falla si una función retorna 'None' sin indicarlo (ej. falta return)
warn_no_return = true

# Falla si hay código "muerto" (unreachable)
warn_unreachable = true

# Excluir carpetas (ej. configuración o migraciones)
exclude = 'src/config/|migrations/'

Java (Checkstyle): El "Contrato" XML (Nivel StyleCop)

Como mencionó, Checkstyle es el análogo a StyleCop. Es un motor que ejecuta reglas definidas en XML. Es 100% configurable.

La clave es el módulo "TreeWalker", que camina por el Árbol de Sintaxis Abstracto (AST) y aplica reglas a los nodos.

<!-- Fichero: checkstyle.xml -->
<module name="Checker">
    <property name="charset" value="UTF-8"/>
    
    <!-- Módulo que camina por el AST -->
    <module name="TreeWalker">
    
        <!-- 1. Forzar Regex para nombres de métodos -->
        <!-- (Forzar "camelCase" ej: miMetodo) -->
        <module name="MethodName">
            <property name="format" value="^[a-z][a-z0-9][a-zA-Z0-9_]*$"/>
        </module>

        <!-- 2. Limitar la Complejidad Ciclomática -->
        <module name="CyclomaticComplexity">
            <property name="max" value="10"/>
        </module>

        <!-- 3. Prohibir código específico (ej. System.out) -->
        <module name="RegexpSinglelineJava">
            <property name="format" value="System\.out\.println"/>
            <property name="message" value="Prohibido usar System.out. Usar Logger."/>
        </module>

    </module>
</module>

PHP (phpcs): Extendiendo y Sobrescribiendo Reglas

En PHP, la práctica habitual es extender un estándar base (como PSR-12) y luego sobrescribir las reglas con las que el equipo no está de acuerdo.

Esto ofrece un balance perfecto entre seguir el estándar de la industria y permitir la personalización.

<?xml version="1.0"?>
<ruleset name="MiEstandarDeProyecto">
    <description>Nuestro estándar, basado en PSR-12.</description>

    <!-- 1. Cargar el estándar base PSR-12 -->
    <rule ref="PSR12"/>

    <!-- 2. Sobrescribir una regla de PSR-12 -->
    <!-- (Ej. No estamos de acuerdo con el límite de 120 chars) -->
    <rule ref="Generic.Files.LineLength">
        <properties>
            <!-- Lo bajamos a 100 -->
            <property name="lineLimit" value="100"/>
            <property name="absoluteLineLimit" value="100"/>
        </properties>
    </rule>

    <!-- 3. Añadir una regla que no está en PSR-12 -->
    <!-- (Ej. Forzar que los nombres de clase sean UpperCamelCase) -->
    <rule ref="Squiz.Classes.ValidClassName.NotCamelCaps"/>
    
    <!-- 4. Excluir reglas específicas -->
    <rule ref="PSR1.Methods.CamelCapsMethodName.NotCamelCaps">
        <exclude-pattern>*/tests/*</exclude-pattern> <!-- Permitir test_names() en tests -->
    </rule>
</ruleset>

Conclusión

El "estilo" se define en estos ficheros de configuración, que actúan como el contrato de calidad del proyecto, forzado automáticamente por el pipeline de CI/CD.
