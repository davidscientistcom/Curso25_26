El Problema: El Caos (Antes de 2016)

Antes de `pyproject.toml`, la configuración de un proyecto de Python era un caos:

1. El Problema de `setup.py`

  • Era un fichero ejecutable, no un fichero de datos estático.
  • Esto creaba un problema de "huevo y gallina":
    • `pip` no podía saber qué dependencias (ej. `setuptools`) necesitaba el `setup.py`... ¡sin ejecutar el `setup.py`!
  • Peligroso (ejecutar código arbitrario para instalar).

2. El Problema de los "Dotfiles"

  • Cada herramienta de calidad inventó su propio fichero de configuración.
  • La raíz del proyecto se llenaba de ruido:
    • .flake8
    • .pylintrc
    • mypy.ini
    • .isort.cfg
    • pytest.ini

La Solución: Dos Estándares (PEPs)

La comunidad de Python resolvió esto con dos estándares que funcionan juntos. Es crucial entender la diferencia.

PEP 518: El "Manifiesto"

Qué hace: Introduce el fichero `pyproject.toml`.

Propósito: Define un lugar estándar para declarar las dependencias del sistema de construcción (build).

Responde a la pregunta:
"¿Qué herramientas necesito para *empezar a construir* tu proyecto?"

PEP 517: La "Interfaz"

Qué hace: Define una interfaz (API) que deben seguir esos sistemas de construcción.

Propósito: Desacopla a `pip` (el *frontend*) de `setuptools` (el *backend*).

Responde a la pregunta:
"Ya que tengo tus herramientas, ¿cómo te construyo?"

A Fondo: PEP 518 (El Fichero)

PEP 518 solo estandariza *una* sección dentro de `pyproject.toml`:

[build-system]

Le dice a `pip` qué necesita instalar en un entorno aislado *antes* de intentar construir el paquete.

# Fichero: pyproject.toml

[build-system]

# 1. "Pip, necesitas estas herramientas para construirme"
#    (ej. setuptools, poetry-core, flit-core, etc.)
requires = ["setuptools>=61.0", "wheel"]

# 2. "Una vez instaladas, esta es la función a la que debes llamar"
build-backend = "setuptools.build_meta"

Esto resolvió el problema del "huevo y gallina". `pip` ya no *asume* que `setuptools` existe.

A Fondo: PEP 517 (La Interfaz)

PEP 517 es el estándar que permite a `setuptools` competir con `poetry`, `flit` y otros.

Define una API agnóstica. El `build-backend` (visto en PEP 518) debe implementar funciones específicas que `pip` (el *frontend*) llamará. Las principales son:

build_wheel(...)

El *backend* (ej. `poetry-core`) debe tener esta función. `pip` la llama, y el *backend* es responsable de generar el fichero `.whl` (la "rueda").

build_sdist(...)

El *backend* debe tener esta función para generar la distribución de código fuente (el `.tar.gz`).

Este desacoplamiento es la clave: `pip` no necesita saber *cómo* `poetry` construye una "rueda", solo necesita saber que puede llamar a `poetry.core.masonry.api.build_wheel()`.

El Flujo Completo (¡La Magia!)

Esto es lo que pasa ahora cuando ejecutas pip install mi_paquete:

1. pip (Frontend) descarga el código de `mi_paquete`.
2. pip lee `pyproject.toml` (gracias a PEP 518).
3. pip ve `[build-system]` que dice `requires = ["poetry-core"]`.
4. pip crea un entorno aislado temporal.
5. pip instala `poetry-core` en ese entorno aislado.
6. pip ve `build-backend = "poetry.core.masonry.api"`.
7. pip llama a la función `build_wheel()` de ese backend (gracias a PEP 517).
8. poetry-core (Backend) se ejecuta y genera `mi_paquete-1.0.whl`.
9. pip toma ese `.whl` y lo instala en el entorno final del usuario.

El "Efecto Secundario": Un Hogar para TODO

PEP 518/517 solo estandarizaron el `[build-system]`.

Pero la comunidad de Python vio que por fin había UN fichero de configuración estándar... y todos se subieron al barco.

Se creó un estándar de facto (ahora oficial) para usar la sección [tool.*] para la configuración de todas las demás herramientas.

Ejemplo: Un pyproject.toml Completo

Así se ve un fichero que lo controla todo: Build, Dependencias y Calidad.

# 1. Definición del Proyecto (Estándar de Poetry/Flit)
[tool.poetry]
name = "mi_app"
version = "0.1.0"
dependencies = [ "python = ^3.10", "fastapi = ^0.100.0" ]
dev-dependencies = [ "pytest", "black", "mypy" ]

# 2. El Sistema de Build (PEP 518 / 517)
[build-system]
requires = ["poetry-core>=1.0.0"]
build-backend = "poetry.core.masonry.api"

# 3. Configuración de Herramientas (El "Efecto Secundario")
[tool.black]
line-length = 88

[tool.mypy]
strict = true
ignore_missing_imports = true

[tool.pytest.ini_options]
pythonpath = ["src"]

Conclusión

PEP 518/517 son los estándares que permitieron a Python desacoplar la construcción de paquetes y, como efecto secundario, nos dieron `pyproject.toml` como el fichero de manifiesto único para gobernar nuestros proyectos.
