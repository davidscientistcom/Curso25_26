Testing de Software en Python

De la Teoría a la Práctica: Cobertura, Cohesión y Acoplamiento.

¿Por Qué Probar el Código?

Antes de escribir pruebas, debemos entender qué hace que el código sea "probable" y fácil de mantener.

Métrica Clave: Cobertura de Código

¿Qué es la Cobertura?

Es una métrica (en %) que mide cuántas líneas de tu código fuente se ejecutan durante la batería de pruebas.

Una alta cobertura (>80%) no garantiza código libre de errores, pero una baja cobertura (<50%) sí garantiza que tienes partes del código sin probar.

Herramienta: coverage.py

Es la herramienta estándar en Python para medir la cobertura.

# Instalar
$ pip install coverage

# Ejecutar (en lugar de 'pytest')
$ coverage run -m pytest

# Ver reporte en terminal
$ coverage report -m

# Ver reporte en HTML
$ coverage html

Principio de Ingeniería: Cohesión

Mide qué tan enfocada está una unidad de código (módulo, clase, función) en una sola tarea.

Alta Cohesión (¡Bueno!)

Una función que solo calcula un promedio.

Una clase Usuario que solo maneja datos del usuario.

Baja Cohesión (¡Malo!)

Una función que calcula un promedio, Y escribe en un archivo Y envía un email.

Objetivo: Alta Cohesión. Cada pieza de código debe hacer UNA sola cosa bien.

Principio de Ingeniería: Acoplamiento

Mide el grado de dependencia entre dos o más unidades de código.

Alto Acoplamiento (¡Malo!)

El Módulo A conoce los detalles internos del Módulo B. Si cambias B, rompes A.

Son difíciles de probar por separado.

Bajo Acoplamiento (¡Bueno!)

Los módulos se comunican a través de interfaces simples (parámetros y retornos).

Puedes probar A "mockeando" (simulando) B.

Objetivo: Bajo Acoplamiento. El código debe ser modular e independiente.

Caso Práctico: El Código "Malo"

Veamos un código que es difícil de probar debido a su baja cohesión y alto acoplamiento.

Ejemplo: Código No Testeable

report_processor.py (Malo)

Este script hace todo a la vez:

  1. Se conecta directo a la BD (¡Acoplamiento!)
  2. Tiene lógica de negocio (filtrado).
  3. Formatea los datos.
  4. Escribe en un archivo.

Problemas:

  • ¿Cómo lo pruebas sin una BD real?
  • ¿Cómo pruebas el formato sin la lógica?
  • (Baja Cohesión y Alto Acoplamiento)
# report_processor.py (Malo)
import sqlite3

def process_user_report(min_age):

    # 1. Acoplamiento alto (BD)
    connection = sqlite3.connect("prod.db")
    cursor = connection.cursor()

    # 2. Lógica de negocio
    cursor.execute(
        "SELECT name, email, age FROM users"
    )

    final_report = "USER REPORT\n==========\n"

    for name, email, age in cursor.fetchall():

        # 3. Más lógica
        if age >= min_age:

            # 4. Formateo (Baja Cohesión)
            final_report += f"User: {name.upper()} ({email})\n"
    
    # 5. Efecto secundario
    with open("report.txt", "w") as f:
        f.write(final_report)

Pruebas de Caja Blanca

La complejidad (3) nos dice que necesitamos 3 casos de prueba para una cobertura de caminos (Basis Path Testing).

Caminos a probar:

  1. El for no se ejecuta (0 usuarios).
  2. El for se ejecuta, pero el if es Falso (usuarios < min_age).
  3. El for se ejecuta, y el if es Verdadero (usuarios >= min_age).

Refactorización: El Código "Bueno"

Aplicamos técnicas para aumentar la cohesión y reducir el acoplamiento.

Técnicas de Refactorización Aplicadas

  • Principio de Responsabilidad Única (Alta Cohesión): Separamos la lógica en funciones pequeñas y enfocadas (obtener datos, filtrar datos, formatear datos).
  • Inyección de Dependencias (Bajo Acoplamiento): En lugar de *crear* la conexión a la BD, la *recibimos* como parámetro (db_cursor).
  • Evitar Efectos Secundarios: La función principal *retorna* el reporte en lugar de escribirlo en un archivo. Otra parte del sistema se encargará de guardarlo.

Ejemplo: Código Testeable

report_processor_v2.py (Bueno)

¡Ahora podemos probar cada parte por separado!

  1. Probar _format_report_line aisladamente.
  2. Probar process_report pasando una lista falsa (mock).
  3. Probar fetch_users pasando un cursor falso.

¡El código ahora es testeable!

La complejidad ciclomática de cada función es ahora 1 o 2, mucho más simple.

# report_processor_v2.py (Bueno)

# 1. Responsabilidad Única (Formato)
def _format_report_line(user_data):
    name, email = user_data['name'], user_data['email']
    return f"User: {name.upper()} ({email})\n"

# 2. Responsabilidad Única (Lógica)
# 3. Inyección de Dependencias (users)
def process_report(users: list, min_age: int):
    report_lines = []
    for user in users:
        if user['age'] >= min_age:
            report_lines.append(
                _format_report_line(user)
            )
    
    if not report_lines:
        return "No users found."
    
    return "".join(report_lines)

# 4. Responsabilidad Única (Datos)
# 5. Inyección de Dependencias (cursor)
def fetch_users(db_cursor):
    db_cursor.execute(
        "SELECT name, email, age FROM users"
    )
    # Devuelve datos crudos, no procesa
    return [
        {"name": n, "email": e, "age": a}
        for n, e, a in db_cursor.fetchall()
    ]
