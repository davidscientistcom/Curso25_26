De Monstruo a Modelo

Caso Práctico: Refactorizando y Testeando un Código "Intesteable"

El Problema: "Código Monstruoso"

Nuestro punto de partida, codigo_monstruoso.py, era imposible de probar por diseño.

¿Qué lo hacía "Intesteable"?

  • Alto Acoplamiento: Dependía *directamente* de requests, datetime y el sistema de ficheros.
  • Baja Cohesión: Mezclaba responsabilidades: lógica de negocio, llamadas a API, formateo de strings y escritura a disco.
  • Efectos Secundarios: Modificaba un estado global (USUARIOS_PROCESADOS_GLOBALMENTE) y escribía ficheros.
  • Dependencias Impredecibles: Usaba datetime.now(), haciendo que el output nunca fuera idéntico.
  • Problemas de Seguridad: Tenía una API Key "hardcodeada" (¡detectada por bandit!).

El Plan de Ataque: Nuestro Toolbox

Paso 1: Limpieza

Formateo y estilo.

  • flake8
  • isort
  • black

Paso 2: Análisis

Detectar problemas graves.

  • bandit (Seguridad)
  • mypy (Tipos)

Paso 3: Refactorización

Arreglar el diseño.

  • Inyección de Dependencias
  • pytest
  • pytest-mock

Pasos 1 y 2: Limpieza y Análisis

Limpieza

isort ordenó los imports.

black formateó todo el código.

flake8 nos ayudó a eliminar imports y variables no usadas (como os y sys al principio).

Resultado: El código era más legible, pero seguía siendo "intesteable".

Análisis

bandit encontró la API Key hardcodeada.

Solución: Mover la key a una variable de entorno (os.environ.get).

mypy encontró que pasábamos un str ("diez") a una función que esperaba int.

Solución: Corregir los datos de entrada y añadir type hints más estrictos (List[int]).

Paso 3: La Gran Refactorización

  • Concepto Clave: Inyección de Dependencias. En lugar de *crear* dependencias (como requests o datetime.now()), la función debe *recibirlas* como parámetros.
  • Acción 1: Extraer Lógica Pura. Creamos calcular_descuento_usuario(). Esta función no tiene I/O, no usa requests, solo recibe datos y devuelve datos. Es 100% testeable.
  • Acción 2: Abstraer la API. Creamos una clase ClienteAPI. Esta clase es la *única* que sabe de requests.
  • Acción 3: Convertir el "Monstruo". La función principal ahora es una "Orquestadora". Recibe el ClienteAPI y la fecha_actual como parámetros.
  • Acción 4: Mover Efectos Secundarios. El if __name__ == "__main__": es ahora el único lugar "sucio" que crea las dependencias (ClienteAPI real, datetime.now()) y escribe el fichero.

El Resultado: Diseño Limpio y Cohesivo

Lógica Pura (Testeable)

def calcular_descuento_usuario(
    usuario: Dict[str, Any]
) -> Tuple[Dict[str, Any], str]:

Recibe un diccionario, devuelve un diccionario y un string. Simple, puro, sin dependencias.

Orquestador (Testeable)

def procesar_usuarios_y_reportar_refactorizado(
    lista_ids_usuarios: List[int],
    cliente_api: ClienteAPI,
    fecha_actual: datetime,
) -> Tuple[str, List[Dict[str, Any]]]:

Recibe la lógica "sucia" (API, fecha) como parámetros, permitiéndonos inyectar *Mocks* en los tests.

Paso 4: ¡Ahora SÍ Podemos Testear!

Test Unitario (Lógica Pura)

Para calcular_descuento_usuario, no necesitamos mocks. Solo pasamos datos y comprobamos el resultado.

def test_calcular_descuento_pais_es():
    usuario_es = {"id": 1, ...}
    usuario_upd, log = calcular_descuento...(usuario_es)
    assert usuario_upd['descuento_aplicado'] == 0.2
    assert "descuento ES" in log

Test de Integración (Orquestador)

Usamos pytest-mock (vía mocker o MagicMock) para simular el ClienteAPI y la fecha_actual.

def test_procesar_reporte_con_mocks(mocker):
    # 1. Preparar Mocks
    fecha_fija = datetime(2025, 1, 1)
    mock_cliente = MagicMock(spec=ClienteAPI)
    mock_cliente.obtener_usuario.side_effect = [
        {"id": 1, ...}, # Caso 1
        requests.HTTPError(...) # Caso 2
    ]

    # 2. Ejecutar
    reporte, _ = procesar_usuarios_...(
        [1, 99], mock_cliente, fecha_fija
    )

    # 3. Verificar
    assert "Usuario 1 tiene descuento" in reporte
    assert "Usuario 99 no encontrado" in reporte

Paso 5: Medir la Cobertura

Ejecutamos pytest --cov=codigo_refactorizado.

El resultado: ¡Casi 100% de cobertura!

Ahora tenemos confianza en que nuestra lógica de negocio (la parte más importante) está probada para todos los casos: descuentos de ES, de MX, menores de edad, y también probamos los fallos de la API (404, Timeout).

Resumen: Antes vs. Después

ANTES (Monstruo)

  • Monolítico
  • Acoplado (requests, datetime)
  • Efectos Secundarios (ficheros, print)
  • Estado Global
  • Inseguro (API Key)
  • Impredecible (datetime.now)
  • Cobertura de Test: 0% (Intesteable)

DESPUÉS (Modelo)

  • Modular
  • Desacoplado (Inyección de Deps)
  • Lógica Pura + Orquestador
  • Sin Estado Global
  • Seguro (Variables de Entorno)
  • Predecible (Fecha inyectada)
  • Cobertura de Test: 99%+ (Testable)

¿Preguntas?

El código limpio no es solo "más bonito", es una necesidad para la mantenibilidad y la fiabilidad.
