Tu Flujo: `git clone` -> `docker build`

¿Por qué `pyproject.toml` es VITAL para los Servicios en Docker (y no solo para Librerías)?

La Suposición (¡Muy Lógica!)

"Entiendo PEP 518/517 para construir librerías y publicarlas en PyPI..."

PERO...

"...Yo no hago librerías. Yo construyo SERVICIOS."

"Mi flujo es: git clone, docker build, docker run."

Esta distinción es clave. Y tienes razón, tu flujo es el estándar de la industria para servicios (APIs, workers, etc.).

La respuesta: `pyproject.toml` resuelve dos problemas críticos que tu flujo de Docker tiene.

Los 2 Problemas Críticos de tu Flujo

1. El Problema de la Reproducibilidad

Tu `Dockerfile` "antiguo" usa `requirements.txt`.

Si `requirements.txt` dice fastapi>=0.100, hoy instalará la `0.100` y mañana la `0.101`.

¡Tu build de Docker no es determinístico! Es una "bomba de tiempo" que puede romperse en cualquier momento sin que cambies tu código.

2. El Problema del "Build Ciego"

Tu `Dockerfile` "antiguo" asume que el código que clonaste de Git... ¡funciona!

¿Qué pasa si alguien subió código que rompe los tests? ¿O que no pasa `flake8`? ¿O que tiene un fallo de seguridad detectado por `bandit`?

¡Tu `docker build` tendrá éxito y crearás una imagen ROTA sin saberlo!

El Flujo "Antiguo" (El Problema)

Este `Dockerfile` sufre de ambos problemas.

FROM python:3.10-slim

WORKDIR /app
COPY . .

# 1. EL PROBLEMA DE REPRODUCIBILIDAD
#    - ¿Qué versión de fastapi se instalará HOY?
#    - ¿Y mañana? ¿Funcionará?
RUN pip install -r requirements.txt

# 2. EL PROBLEMA DEL "BUILD CIEGO"
#    - Estamos copiando código (COPY .) que
#      podría estar roto, no pasar tests,
#      o tener fallos de linting.
#    - ¡Y lo vamos a ejecutar igualmente!
CMD ["python", "src/main.py"]

Solución 1: El Manifiesto de Dependencias

pyproject.toml + un fichero `.lock` (ej. `poetry.lock` o `pdm.lock`) reemplazan a `requirements.txt`.

¿Qué es un fichero `.lock`?

Es un "manifiesto" de build 100% determinístico. Contiene las versiones exactas de CADA dependencia (y sub-dependencia) con sus hashes.

Tu `pyproject.toml` dice "quiero `fastapi`".

Tu `poetry.lock` dice "para eso, te instalaré `fastapi-0.100.1` (hash: ...), `starlette-0.30.0` (hash: ...), `pydantic-2.1.0` (hash: ...), etc."

¡Esto resuelve el Problema 1! Tu build ahora es 100% reproducible.

El "Nuevo" Dockerfile (Reproducible)

Usando `pyproject.toml` y `poetry` (una herramienta que usa el fichero).

FROM python:3.10-slim
WORKDIR /app

# Instalar Poetry
RUN pip install poetry
RUN poetry config virtualenvs.create false

# --- ¡LA CLAVE ESTÁ AQUÍ! ---
# 1. Copiamos SÓLO los manifiestos
COPY pyproject.toml poetry.lock* ./

# 2. Instalamos desde el LOCK FILE
#    Garantiza un build 100% determinístico.
#    --no-root: no instales "mi_app" (aún no está)
#    --no-dev: no instales pytest, flake8, etc. (son para CI, no runtime)
RUN poetry install --no-root --no-dev

# 3. Copiamos el código fuente (DESPUÉS de las dependencias)
#    (¡Esto mejora el cacheo de capas de Docker!)
COPY src/ ./src/

CMD ["poetry", "run", "python", "src/main.py"]

Solución 2: El Manifiesto de Calidad

¡El mismo `pyproject.toml` que clonaste de Git resuelve el Problema 2!

Contiene el "Contrato de Calidad" en las secciones `[tool.*]`.

# Fichero: pyproject.toml

# ... (sección de dependencias [tool.poetry]) ...

# --- EL CONTRATO DE CALIDAD ---

[tool.flake8]
max-line-length = 88
ignore = "E203, W503"

[tool.mypy]
strict = true
ignore_missing_imports = true

[tool.bandit]
skips = ["B101"]

[tool.pytest.ini_options]
pythonpath = ["src"]

Esto le dice a tu sistema de CI (GitHub Actions, Jenkins) *exactamente* qué reglas forzar.

El Flujo Profesional (CI + Docker)

Tu `docker build` no se ejecuta "a ciegas". Se ejecuta *después* de que un pipeline de CI haya validado el "Contrato de Calidad".

Tu Flujo (`git clone` -> `docker build`) se convierte en:

git push
    ↓
GitHub Actions (CI)
    ↓
1. poetry install (Instala TODO, incl. `pytest`)
2. poetry run flake8 src (Lee [tool.flake8])
3. poetry run mypy src   (Lee [tool.mypy])
4. poetry run pytest     (Lee [tool.pytest])
    ↓
SI TODO PASA...
    ↓
docker build (Usando el `Dockerfile` reproducible)
    ↓
docker push (A la registry)

Conclusión: Los 2 Roles del Manifiesto

En tu flujo de Servicio Docker, el `pyproject.toml` NO es para "construir librerías".

Es el **Manifiesto Único de tu Proyecto** que cumple dos roles:

Rol 1: Manifiesto de Build

Para: Docker (y `poetry`)

Qué usa: `[tool.poetry.dependencies]` + `poetry.lock`.

Resultado: Un build 100% reproducible y determinístico. (Resuelve Problema 1).

Rol 2: Manifiesto de Calidad

Para: El Pipeline de CI (y `poetry`)

Qué usa: `[tool.flake8]`, `[tool.mypy]`, `[tool.pytest]`.

Resultado: Un "Quality Gate" que impide que código roto llegue a producción. (Resuelve Problema 2).

Resumen

pyproject.toml es el "plano" (calidad) y la "lista de materiales" (dependencias) de tu servicio.
