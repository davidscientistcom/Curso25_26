Ecosistema de Testing y Calidad

Una guía de las herramientas clave para un código Python robusto y mantenible.

1. Frameworks de Testing

Los "Jefes". Las herramientas fundamentales para escribir y ejecutar pruebas.

pytest

Qué hace:

Es el framework estándar de facto. Potente, fácil de usar (usa assert simples) y con un ecosistema de plugins gigante (pytest-cov, pytest-mock).

Instalación:

$ pip install pytest
# test_mi_codigo.py
def sumar(a, b):
    return a + b

# Un test de pytest es solo una función
def test_sumar():
    # Usa 'assert' simple, sin self.assertEqual
    assert sumar(2, 2) == 4
    assert sumar(5, 0) == 5

# --- Ejecución ---
$ pytest
================ test session starts ================
test_mi_codigo.py .                               [100%]
================ 1 passed in 0.01s ================

unittest

Qué hace:

Es la librería de testing que viene integrada por defecto en Python. Es más verbosa (usa self.assertEqual()) y está inspirada en los frameworks xUnit de Java.

Instalación:

# (Viene con Python, no requiere instalación)
# test_mi_codigo_unit.py
import unittest

def sumar(a, b):
    return a + b

# Requiere crear una clase que hereda
class TestSuma(unittest.TestCase):
    
    # Los tests deben empezar con 'test_'
    def test_sumar_positivos(self):
        # Usa métodos 'assert' específicos
        self.assertEqual(sumar(2, 2), 4)

# --- Ejecución ---
$ python -m unittest test_mi_codigo_unit.py

hypothesis

Qué hace:

Testing Basado en Propiedades. En lugar de escribir 10 tests, defines 1 "propiedad" (una regla) y hypothesis genera cientos de datos "difíciles" (casos borde) para intentar romper tu código.

Instalación:

$ pip install hypothesis
# test_con_hypothesis.py
from hypothesis import given
from hypothesis.strategies import integers

def sumar(a, b):
    return a + b

# 'given' genera datos de prueba
@given(a=integers(), b=integers())
def test_propiedad_suma(a, b):
    # Definimos una "propiedad" o regla:
    # La suma debe ser conmutativa.
    assert sumar(a, b) == sumar(b, a)

# --- Ejecución ---
$ pytest
...
================ 1 passed in 0.20s ================
(Hypothesis ejecutó cientos de ejemplos)

2. Calidad de Código

Linters (detectan errores) y Formateadores (corrigen estilo).

flake8

Qué hace:

Es un linter. Revisa tu código buscando errores lógicos (ej. variables no usadas) y errores de estilo (ej. no cumples la guía PEP 8). Es el más popular y un gran punto de partida.

Instalación:

$ pip install flake8
# mi_codigo_sucio.py
import os # Error: Import no usado

def mi_funcion( a,b ): # Error: Mal espaciado
    variable_no_usada = 1
    return a+b # Error: Falta espacio

# --- Ejecución ---
$ flake8 mi_codigo_sucio.py

mi_codigo_sucio.py:3:1: E302 expected 2 blank lines, found 1
mi_codigo_sucio.py:3:19: E231 missing whitespace after ','
mi_codigo_sucio.py:4:5: F841 local variable ... unused
mi_codigo_sucio.py:5:11: E225 missing whitespace around operator

pylint

Qué hace:

Un linter mucho más estricto y configurable que flake8. Analiza el código en profundidad (ej. detecta código duplicado) y te da una nota de calidad sobre 10.

Instalación:

$ pip install pylint
# mi_codigo_sucio.py (El mismo de antes)
import os 

def mi_funcion( a,b ): 
    variable_no_usada = 1
    return a+b

# --- Ejecución ---

black

Qué hace:

Es un formateador de código. No te avisa de los errores de estilo, sino que directamente reformatea tu código. Su lema es "no hay nada que discutir", impone un estilo único.

Instalación:

$ pip install black
# mi_codigo.py (ANTES)
def mi_funcion( a,b ): 
    return {'key':a, 'long_key':
    'un valor muy largo que no cabe'}

# --- Ejecución ---
$ black mi_codigo.py
reformatted mi_codigo.py

# mi_codigo.py (DESPUÉS)
def mi_funcion(a, b):
    return {
        "key": a,
        "long_key": "un valor muy largo que no cabe",
    }

isort

Qué hace:

Un formateador específico para tus import. Los ordena alfabéticamente y los agrupa (librerías estándar, librerías de terceros, tu código).

Instalación:

$ pip install isort
# mi_codigo.py (ANTES)
import os
import pytest
from mi_app import models
import sys
from mi_app.views import admin_view
import requests

# --- Ejecución ---
$ isort mi_codigo.py
Fixing mi_codigo.py

# mi_codigo.py (DESPUÉS)
import os
import sys

import pytest
import requests

from mi_app import models
from mi_app.views import admin_view

3. Cobertura de Tests

Mide qué porcentaje de tu código es ejecutado por tus tests.

coverage.py

Qué hace:

Mide la cobertura de tus tests. Te dice qué líneas se han ejecutado y cuáles no. pytest-cov es el plugin que lo integra fácilmente con pytest.

Instalación:

$ pip install coverage pytest-cov
# --- Ejecución (Forma fácil con pytest) ---
$ pytest --cov=mi_proyecto
================ test session starts ================
... (tests) ...

----------- coverage: platform ... -----------
Name                   Stmts   Miss  Cover
--------------------------------------------
mi_proyecto/__init__.py      1      0   100%
mi_proyecto/app.py          20      2    90%
mi_proyecto/utils.py         8      4    50%
--------------------------------------------
TOTAL                     29      6    79%

# --- Generar reporte HTML (más visual) ---
$ pytest --cov=mi_proyecto --cov-report=html
(Abrirá un fichero 'htmlcov/index.html')

4. Simulación y Mocks

La herramienta clave para el desacoplamiento y los tests unitarios aislados.

unittest.mock

Qué hace:

Te permite crear "Mocks" u "objetos falsos". Simula (falsea) una base de datos, una API externa o un objeto complejo para que tus tests unitarios sean aislados y rápidos. pytest-mock lo integra con pytest.

Instalación:

$ pip install pytest-mock
# app.py
import requests
def get_precio_bitcoin():
    # ¡Llamada a una API externa! (Lento, inestable)
    r = requests.get("https://api.coindesk.com/...")
    return r.json()['bpi']['USD']['rate_float']

# test_app.py
from app import get_precio_bitcoin

def test_get_precio_bitcoin(mocker): # Inyectado por pytest-mock
    # 1. Preparamos un Mock para la respuesta
    mock_response = mocker.Mock()
    mock_response.json.return_value = {
        'bpi': {'USD': {'rate_float': 50000.0}}
    }
    
    # 2. "Parcheamos": reemplazamos 'requests.get'
    mocker.patch("requests.get", return_value=mock_response)
    
    # 3. El test no llama a la API real
    assert get_precio_bitcoin() == 50000.0

5. Seguridad

Encuentra vulnerabilidades antes de que lleguen a producción.

bandit

Qué hace:

Un analizador de seguridad estático. Escanea tu código buscando vulnerabilidades comunes, como contraseñas "hardcodeadas", uso de eval(), inyecciones SQL, etc.

Instalación:

$ pip install bandit
# app_insegura.py
import subprocess

def ejecutar_comando(cmd):
    # Vulnerabilidad: Inyección de shell
    subprocess.call(cmd, shell=True)

# Vulnerabilidad: Contraseña hardcodeada
password = "MySuperSecretP4ss!"

# --- Ejecución ---
$ bandit -r .
[B404] subprocess_call: ... (shell=True)
   ...
   Severity: High Confidence: High

[B105] hardcoded_password_string: ...
   ...
   Severity: Medium Confidence: High

6. Verificación de Tipos

Añade la seguridad de los lenguajes estáticos a Python.

mypy

Qué hace:

Es el verificador de tipos (type checker) estándar. Lee tus "type hints" (ej. nombre: str) y te avisa antes de ejecutar si estás pasando un tipo de dato incorrecto.

Instalación:

$ pip install mypy
# app_con_tipos.py
def saludar(nombre: str) -> str:
    return "Hola, " + nombre

def sumar(a: int, b: int) -> int:
    return a + b

# --- Errores detectados ---
saludar(123)
sumar("1", "2")

# --- Ejecución ---
$ mypy app_con_tipos.py
app_con_tipos.py:9: error: Argument 1 to "saludar" 
    has incompatible type "int"; expected "str"
app_con_tipos.py:10: error: Argument 1 to "sumar" 
    has incompatible type "str"; expected "int"

7. Orquestación

Herramientas para ejecutar todas las demás de forma automática y consistente.

pre-commit

Qué hace:

Una herramienta para gestionar Git Hooks. Te permite ejecutar automáticamente black, flake8, mypy, etc., cada vez que haces git commit. Si alguna herramienta falla, el commit se detiene.

Instalación:

$ pip install pre-commit
# .pre-commit-config.yaml (En la raíz de tu repo)
repos:
-   repo: https://github.com/psf/black
    rev: 24.4.2
    hooks:
    -   id: black
-   repo: https://github.com/PyCQA/flake8
    rev: 7.1.0
    hooks:
    -   id: flake8
-   repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
    -   id: mypy

# --- Instalación en tu repo ---
$ pre-commit install
pre-commit installed at .git/hooks/pre-commit

# (Ahora, 'git commit' fallará si black, flake8 o mypy fallan)

tox

Qué hace:

Una herramienta de automatización que prueba tu código en múltiples entornos virtuales limpios. Se usa para asegurar que tu librería funciona correctamente con diferentes versiones de Python (ej. 3.9, 3.10, 3.11).

Instalación:

$ pip install tox
# tox.ini (o en pyproject.toml)
[tox]
# Lista de entornos a probar
envlist = py39, py310, py311, flake8

[testenv]
# Comandos a ejecutar para cada entorno de python
deps = pytest
commands = pytest

[testenv:flake8]
# Entorno solo para el linter
deps = flake8
commands = flake8 mi_proyecto

# --- Ejecución ---
$ tox
... (crea 4 entornos virtuales y ejecuta tests) ...
  py39: commands succeeded
  py310: commands succeeded
  py311: commands succeeded
  flake8: commands succeeded
  congratulations :)
