1. Definición Formal y Paradigma

El Cloud Computing no es simplemente "el ordenador de otro". Es un cambio de paradigma basado en la abstracción y la orquestación.

  • Definición Técnica: Disponibilidad bajo demanda de recursos del sistema informático (almacenamiento y potencia de cómputo) sin gestión activa directa por parte del usuario[cite: 305].
  • El "Enabler" Tecnológico: La virtualización. Un servidor físico se divide en múltiples servidores virtuales mediante un hipervisor, eliminando los límites físicos del hardware[cite: 308, 309].
  • Modelo Económico: Transición de CAPEX (Gasto de Capital - Inversión inicial) a OPEX (Gasto Operativo - Pago por uso)[cite: 4, 306].
Concepto Clave: El proveedor (AWS, Azure) gestiona la "física" (centros de datos, refrigeración, seguridad física), mientras el usuario consume "servicios abstractos" (instancias, buckets, lambdas).

2. Ventajas Arquitectónicas y Operativas

Más allá del ahorro, el Cloud ofrece capacidades imposibles en on-premise tradicional:

  • Elasticidad vs. Escalabilidad:
    Escalabilidad: Capacidad de crecer (añadir recursos)[cite: 20].
    Elasticidad: Capacidad de ajustar recursos automáticamente (arriba y abajo) según la demanda en tiempo real, evitando el aprovisionamiento excesivo[cite: 318].
  • Agilidad de Despliegue (Time-to-market): Aprovisionamiento de infraestructura en minutos mediante APIs, en lugar de semanas de compra e instalación física[cite: 317].
  • Alta Disponibilidad y Resiliencia: Redundancia geográfica automática. Si un servidor falla, el control pasa a otro sin interrupción perceptible[cite: 26, 243].
  • Sostenibilidad: Economía de escala en eficiencia energética (PUE optimizado) superior a un CPD corporativo medio[cite: 322].

3. Desafíos y Consideraciones Críticas

Como ingenieros, debemos evaluar los trade-offs:

  • Latencia de Red: Limitaciones físicas de la velocidad de la luz. Sistemas críticos en tiempo real pueden sufrir retrasos si el CPD está lejos (e.g., clínica en España, datos en Alemania)[cite: 328, 330].
  • Dependencia del Proveedor (Vendor Lock-in): El uso de servicios propietarios (ej. AWS DynamoDB, Azure Functions) crea un acoplamiento fuerte. Migrar requiere reingeniería, no solo copiar datos[cite: 331, 333].
  • Seguridad y Soberanía del Dato: Aunque la nube es segura, la externalización de datos sensibles (bancarios, salud) choca con regulaciones estrictas. El "Modelo de Responsabilidad Compartida" es vital[cite: 335, 336].
  • Interrupción de Servicio: Dependencia total de la conectividad a Internet. Sin red, no hay operación[cite: 326].

4.1. IaaS: Infraestructura como Servicio

El equivalente al CPD tradicional, pero virtualizado y alquilado.

  • Qué ofrece: Recursos "crudos" de computación (CPU, RAM), redes (VPC, Balanceadores) y almacenamiento (Bloque/Objeto)[cite: 29, 361].
  • Nivel de Control: Máximo. El usuario administra el Sistema Operativo (parcheo, actualizaciones) hacia arriba.
  • Caso de Uso: Migraciones "Lift and Shift" de aplicaciones legacy, control total de la configuración del kernel, bases de datos personalizadas.
  • Ejemplos: Amazon EC2, Google Compute Engine, Azure Virtual Machines[cite: 30].

4.2. PaaS: Plataforma como Servicio

Abstracción para desarrolladores. Elimina la gestión del S.O. (el "undifferentiated heavy lifting").

  • Qué ofrece: Entornos de ejecución (Runtimes para Java, Python, .NET), bases de datos gestionadas, herramientas de despliegue[cite: 31, 354].
  • Ventaja Clave: El desarrollador solo se preocupa por el código. El proveedor gestiona parches de seguridad del OS, escalado automático y balanceo[cite: 356].
  • Evolución Moderna: Contenedores y Orquestación (Kubernetes gestionado como EKS o AKS) se consideran el PaaS moderno[cite: 357, 358].
  • Ejemplos: Google App Engine, AWS Elastic Beanstalk, Heroku[cite: 32].

4.3. SaaS: Software como Servicio

Software listo para consumir. Modelo de entrega predominante hoy en día.

  • Qué ofrece: Aplicación final accesible vía navegador o API.
  • Nivel de Control: Mínimo. El usuario solo configura la aplicación y gestiona sus datos[cite: 33, 346].
  • Características: Actualizaciones transparentes y automáticas (CI/CD del proveedor), modelo de suscripción[cite: 350].
  • Ejemplos: Google Workspace, Microsoft 365, Salesforce, Dropbox, Slack[cite: 33, 46].

5. Modelos de Despliegue de Nube

¿Dónde reside físicamente la infraestructura y quién más la usa?

Tipo Descripción Técnica Ventaja Principal
Pública Infraestructura multi-tenant (compartida) propiedad del proveedor. Acceso vía Internet público o VPN[cite: 88, 393]. Capex cero, escalabilidad infinita, mantenimiento nulo.
Privada Infraestructura single-tenant (dedicada). Puede ser On-Premise o Hosted (en proveedor)[cite: 89, 397]. Control total, seguridad y cumplimiento normativo estricto.
Híbrida Conexión orquestada entre Nube Pública y Privada. Permite "Cloud Bursting" (desbordar a pública en picos de demanda)[cite: 90, 402]. Flexibilidad y optimización de costes.

Multicloud: Estrategia de usar múltiples proveedores públicos (AWS + Azure) para evitar vendor lock-in y aprovechar servicios específicos de cada uno (ej. IA de Google + Integración MS de Azure)[cite: 108, 416].

6. Anatomía Física de la Nube

La "nube" son edificios físicos interconectados por fibra oscura y cables submarinos.

  • Región: Área geográfica independiente (ej. "Europa (París)" o "us-east-1"). Contiene múltiples Zonas de Disponibilidad. Permite cumplir leyes de residencia de datos[cite: 168, 477].
  • Zona de Disponibilidad (AZ): Uno o más Centros de Datos (CPDs) discretos con alimentación, red y refrigeración redundante. Están separadas físicamente (kms) para aislar fallos (incendios, inundaciones) pero conectadas con latencia ultra-baja (<2ms)[cite: 173, 474].
  • Edge Locations (CDN): Puntos de presencia locales para cachear contenido (Netflix, imágenes) cerca del usuario final y reducir latencia[cite: 236].
Regla de Oro de Arquitectura: Diseñar siempre para el fallo (Design for Failure) desplegando en múltiples AZs[cite: 26].

7. El Mercado de Hyperscalers

El mercado es un oligopolio dominado por tres grandes actores (62% del mercado en 2022)[cite: 141].

  • Amazon Web Services (AWS) - 33%: Pionero (2006). Mayor madurez, mayor cantidad de servicios (+200) y cuota de mercado. Estándar de facto[cite: 146, 447].
  • Microsoft Azure - 21%: Fuerte en el entorno empresarial (Enterprise). Integración nativa con Active Directory, Windows Server y Office 365. Crecimiento agresivo[cite: 153, 450].
  • Google Cloud (GCP) - 8%: Especialización en Big Data, Machine Learning (TensorFlow), Kubernetes (creado por Google) y analítica avanzada[cite: 152, 452].

Otros: Alibaba Cloud (Asia), IBM Cloud (Híbrida/Legacy), Oracle Cloud (Bases de Datos)[cite: 149, 457].

Resumen de la Sesión

  • El Cloud Computing es un modelo de consumo de servicios IT basado en OPEX y elasticidad.
  • Los modelos SPI (SaaS, PaaS, IaaS) definen cuánto gestiona el proveedor y cuánto el ingeniero.
  • La arquitectura debe ser resiliente, aprovechando Regiones y Zonas de Disponibilidad.
  • La seguridad es una responsabilidad compartida.
  • El futuro es Híbrido y Multicloud, buscando lo mejor de cada proveedor.
